Strategy

问题：为什么选择 Dart ？

在谷歌我们已经写出了很多优秀的WEB应用程序，我们通过各种方法努力改进其开发过程，但不尽如人意，我们发现缺少引入一门新的语言。现在时机到了。我们设计 Dart 是为了编写用于需要高的运行表现及适合最新应用开发的开发工具更容易。

问题：Dart语言能否弥补目前WEB开发不足？

我们希望能够弥补所有的不足。这就是 Dart 语言以及 Dart 整个项目的目标。Dart 项目认为WEB开发语言应该有所改变，但是同时我们也希望能够提升DOM和其他的类库以及改善目前我们正在使用的开发工具。

同时，谷歌也认为 JavaScript 能进一步被改进，并且积极的投入到这份工作中来。谷歌希望 WEB 开发变得更棒，如果这是由 JavaScript 带来的，我们同样会非常高兴。

问题：Dart 会转移那些以 JavaScript 为基础的WEB开发团体的工作么？

如果人们喜欢 Dart 并且使用它，从某一方面来看，是的，但是任何一种对现有 WEB 开发的改进不都是这样么？完全重新开始或者100%兼容都不可取，所以人们一般旧的和新的一起用。你可以这样看：谷歌重在做有意义的尝试无论是基于 Dart 还是 JavaScript，选择发展中的 Dart同时继续广泛的使用 JavaScript 及其工具，实现，并且发展语言。我们同时都在做，因为我们认为 Dart 值得这么做。

有很多语言用于 WEB 服务器开发：Python 转移了 Perl 的工作么， Java 取代了 C++ 么？再次，从某一方面来看，是的，但是人们通常考虑健康的环境，多语言好于我们只使用单门语言开发，多语言允许快速转换，这已经通过一种标准的过程取得了效果。此外，每种语言都有其自己的存在价值：Groovy 亦不是与 C++直接竞争。人们面对不同工程权衡，选择不同的语言来应对。最后，我们希望客户端开发者们也可以有这种灵活性。

问题：谷歌打算把 Dart 语言交给标准团体控制么？

是的，我们希望那样。一旦 Dart 语言发展到一定程度并且被大众接受，我们期待标准化过程将作为下一步目标。

问题：到这里，你们将怎样推进 Dart 的发展？

我们计划像控制 V8 项目那样控制 Dart。我们将聆听反馈和问题，并且我们将检查外界 Dart 贡献者的修改。 一个有着很好痕迹记录的贡献者将会为 Dart 仓库提交确认。谷歌工程师也将会在公共仓库中工作，进行可见的修改。

问题：谷歌为什么不一开始就将 Dart 作为一个开放标准公布？

我们通过一般途径取得开放标准的程序语言：某人创建一个清晰的初版，人们尝试它，然后我们使其标准化。开放标准 WEB 平台以这种方式整合小的改进，经其他人的尝试后标准化就实现了：canvas就是一个例子。我们知道这种方式增加了大众参与，但是我们认为这种方式只在某些时候有用，特别的，作为一门编程语言，其由开放团队设计是有风险的。

Haskell是近期成功的被开放社区设计的编程语言，始于1990年。最广泛使用的是COBOL，从Ada演变而来。这不是一个程序语言设计的普通方式。在目前主要的编程语言中，只有六个（数字存在争议）是通过这种方式设计的（其中之一是 ALGOL-68)。

问题：谷歌将如何看待 GWT 的未来 ？

Bruce Johnson 发表在 GWT 的博客 (更进一步的评论在 Google+)： "Dart 和 GWT 都以发展结构化 WEB 编程为目标。事实上，很多开发 GWT 的工程师已经参与到 Dart 项目中来。我们把 Dart 当作 GWT 使命（使网页应用更好的服务于终端用户）的革命性推进 ，并且我们看到了它的潜能。在 Dart 不断发展并准备好迎接其黄金时代，我们预测利用 GWT 的开发团队将会乐于探索 Dart。"

"同时，其他确定的是 GWT 将会继续成为开发优秀 WEB 应用（或者其他像愤怒的小鸟这样的游戏）富有创造性、可靠的途径。谷歌的关键项目一直很依靠 GWT 。并且我们计划继续根据世界范围内的需求改进 GWT（并保持开源）。"

观看我们的谷歌 I/O 2012座谈会 Google Web Toolkit 的历史与未来 获得更多信息。

谷歌为什么不开发一个多语言（包括Dart)的字节码虚拟机？

每一种方法都有长处和短处，但是我们感觉创建一个Dart专用的虚拟机作为Dart的运行环境更合理，主要有以下原因：

  • 谷歌已经拥有一门多语言字节码底层虚拟机（使用位码的PNaCl）.
  • 如果 Dart 被单语言字节码虚拟机所支持将有很大好处, 单语言虚拟机将更简单更快，因为它可以专门为该语言进行优化—例如，一个结构化控制流。这些专门的优化使得实现回收器和优化器更加简单。
  • 一个通用的字节码虚拟机将会很大很慢，因为它包括了很多假设和额外的功能，这些对Dart都是没用的：例如，多线程共享堆。
  • 没有字节码虚拟机是真正通用的；他们都假定给某些种类的语言以特权。单语言虚拟机有更多的空间来提升性能，并能深入优化。请详见 Dart 工程师写的关于虚拟机问题的相关文章

语言

问题：Dart 和 JavaScript 很像么？

既是也不是。Dart 项目认为 JavaScript 能做一些改变以提供更具创造性的软件编程能力、更聪明的编辑器和开发环境，以及促使 WEB 应用程序将会向客户端应用程序一样漂亮及令人喜欢。另外，我们不认为所有都需要改变，没必要就不需要变，不是么？

Dart，像 JavaScript 一样是一门动态类型绑定语言。它添加了可选的编译时类型声明帮助你及早的发现问题。它去除了 JavaScript 的某些特性，例如原型机制和全局对象：这将简化虚拟机提高执行速度，并且使得代码实现和重构更加容易。Dart 添加了许多优秀的功能。例举如下：

  • 用户自定义的操作方法。我们用轻量的、易读的代码更新了 DOM 接口。.
  • 匿名函数语法更轻量级化。你在 WEB 开发中经常使用他们；现在他们看起来更棒了。他们为this提供了正确的约束条件和全部的block-level 词汇范围，no gotchas。

Dart 的目标是比 JavaScript 更快，更规则，并且更适合大型程序开发。

问题：Dart 跟 Java 很像么？

好吧， Java 是静态类型绑定的，Dart是动态类型绑定的。Dart拥有可选的静态类型绑定注释，而在 Java 中他们是一种规定。对此这两门语言内在存在着巨大的不同。Dart 是一种 curly-brace 语言，它与 Java 共有一些关键字，像extends、final,因此我们能够理解为什么人们把这两门语言做比较。事实上，我们喜欢明确的、熟悉的语法,学起来会很简单，即使这不能让人感到兴奋。

问题：Dart 与使用闭包函数编译程序的 JavaScript 对比

可选类型注释的想法相类似。Dart 在语法结构上很漂亮。

比较下面的闭包函数编译程序代码：

{% prettify dart %} // 闭包编译程序代码 /** * @param {String} name * @return {String} */ makeGreeting = function(name) { /** @type {String} */ var greeting = 'hello ' + name; return greeting; } {% endprettify %}

以下是 Dart 程序代码：

{% prettify dart %} // Dart 代码 String MakeGreeting(String name) { String greeting = 'hello $name'; return greeting; } {% endprettify %}

如果你正在使用闭包函数并希望转换到 Dart上来，你将会喜欢这种改变。

问题：Dart 与 CoffeeScript 比较

Dart 和 CoffeeScript 都是受 JavaScript 的启迪，并且都能被转换成 JavaScript 的代码。他们是两种不同的选择，尤其在语法风格方面。作为一门语言，公平的说 Dart 与 JavaScript对比比 CoffeeScript 与 JavaScript 对比在语义上改变更大；这将导致更少的一行一行转换，即使这样我们仍坚信 Dart生成的 JavaScript 代码将会更精简并且更快。

如果你喜欢 CoffeeScript 因为它比原生的 JavaScript更结构化，那么你将会喜欢 Dart 的可选静态类型注释。

问题：谷歌是怎么考虑 TypeScript 的？

应用程序的开发。然而，他们采用的方法却完全不同。TypeScript 对 JavaScript 保持后向兼容，然而 Dart 对 JavaScript 的语法及语义做了一定的改变，以根除众多种类的缺陷，并且可以提升性能。web 开发一直只有很少的选择，我们认为 Dart 和 TypeScript 都把目标定为 web 开发者拥有更好的未来！你可以访问我们的博客阅览 更多完成的答复

问题：Dart 与 Go 关系？

都是谷歌启动的语言项目，但是他们是不相干的，并且有各自的目标。这就导致他们作出不同的选择，两门语言有着非常不同的特点，即使我们都尽量去向对方项目组学习。

问题：为什么 Dart 不模仿 Haskell / Smalltalk / Python / Scala / 或其他语言？

各种各样的原因，由具体的语言而定：

对那些与 JavaScript 大有不同的语言：能将 Dart 代码编译成 JavaScript 代码是非常重要的。我们在 GWT 中得到的经验是如果源语言代码如果与 JavaScript 代码差异性太大，它将创建几种情况，这时就需要创建复杂的转换代码，用以模仿源语言代码的行为。这可能导致运行出现偏差，并且开发人员极难发现。

对于那些非主流的语言：我们预测如果 Dart 模仿他们将会导致很难推进其发展。我们的团队拥有这些语言的粉丝，我们可能会受到引诱：采用那些很酷的语言的特点的话，并且将他们深入到语言的发展之中，但是事实上我们认为发明这样的新语言是很费力的。

对于那些比 Dart 更动态的语言：Dart 谨慎的移除了这种随意的运行时修改，以带来更好的性能和优秀的工具。

问题：为什么 Dart 的语法不那么令人兴奋？

我们添加了一下优秀的语法特征，例如this.构造器参数，但是我们一致赞同 Dart 选择熟悉的语法更好。下面是一个团队成员个人见证：“我希望 Dart 拥有更多酷炫的特点，但是我不能否认当我开始用它写代码起，我是很有成效的。”

问题：如果没有 eval()函数和运行时动态类型绑定，Dart 还能算是动态语言么？

作为最初发布的版本没有这些特点，但Dart的未来版本将考虑添加这样的动态特性。可能不会原封不动的添加问题中提到的特点，但我们希望能提供相类似的功能。当我们看到这些功能被添加后，那么每个人都可以确定这门语言的分类。

对我们重要的是你想要用一门动态语言实现什么，并且你可以用 Dart 实现而不感到受约束。你应该能够设计自己的系统，而不会受静态规则的干扰，并在开发或运行过程中进行动态的修改。

因此，举例来说，Dart 是不太可能支持在当前的运行环境中求一个字符串的数值的，但是它可以支持动态加载那种代码到一个新的分离物。dart 是不太可能支持为一个实例值添加字段，但它可能会（通过一个反射系统）支持将字段添加到一个类，并且你可以用noSuchMethod()来添加方法。使用这些功能将会增加运行时的成本，不使用他们进而最小化程序的资源消耗对我们是非常重要的。

这个地方还没有进行开发，所以我们欢迎你提出需要的关于运行时动态特征的想法。

问题：Dart会添加反射么？

我们目前正在进行添加反射功能的开发，它使用mirrors API。

问题：Dart是否会添加元组，模式匹配，非空类型，partial evaluation，optional semicolons...？

这么语言还没有完成。尽管它不能满足所有的需求，但是它可能会增加你提到的这些特性。上诉一些特性不符合这门语言的基本风格，还有一些不能与其他的特点很好的兼容。我们能给与未来开发者最大的礼物是简单易用。

这类问题请查看Dart 已答问题列表，欢迎你在那里提问。将你关于新特点的想法告诉我们。建议列举相关案例代码，用大量代码展示需求的重要性会更好。

如果 Dart 设计人员拒绝了你的建议请不要见怪，特别是目前。移除一个特性比添加一个特性更痛苦，必须谨慎对待，所以 Dart 更需要先添加那些显著需要的特性，后续再考虑其他的。有很多合适的语言特性可以添加到一门单独的语言中而不会把它弄得很杂乱。但是我们仍然非常感谢你的建议和论证。我们希望你可以看到我们谨慎用心的设计选择和公平公开的讨论。

类型

问题：Dart 拥有类型推断么？

类型推断不是一门语言的标准，但是它是一种自由的实现。了解 Dart 拥有动态类型系统是很重要的，但是 Dart 类型推断跟其他语言（像 Haskell）的意义不同。然而，Dart 的编辑器确实做了一些类型推断，例如当你为本地的变量使用 var 。如果对提升性能有用、或者其他原因，我们期待 Dart 虚拟机和 Dart2js 能够使用类型推断。

问题：为什么类型注释是可选的？

我们希望保持 WEB 开发人员熟悉的那种动态类型语言的感觉。强制类型注释不符合这个目标。

问题：为什么类型系统设计的不够健全？

与完全的静态类型系统相比，Dart 拥有一个可选静态注释的动态类型系统。我们这么设计的主要目的是支持工具和文档。我们希望创建一个实用的工具来帮助 WEB 开发人员更顺利的进行开发。特别的，我们希望静态警告可以最优化，而不是动态类型代码是正确的却仍被警告，例如“向下转型”。由于 Dart 执行是类型安全的，我们可以让一些不健全的情况避开静态警告，而在运行时来捕获他们。

典型的面向对象的语言允许向下转型，这仍给类型系统带来不健全并可能导致运行时错误。在 Dart ，我们选择允许不用语法标注的向下转型。

问题：但是你们不需要使用健全的类型信息来提升性能么？

Sound types 能够提升性能，但并不是必须的。我们需要规则、简单的语义。

问题：为什么类型注释对运行时行为没有任何影响？

如果类型注释影响运行时，当用户添加类型信息的时候程序将改变自己的行为，即使程序逻辑仍然保持不变。正常的开发模式是逐步的记录文档并确认，而如果改变了程序的行为，开发人员就没有稳定的基础进行工作了，尤其是在类型可能有错误的情况下。

除此之外，这种策略帮助我们和其他有类型检查功能的工具实现不同的策略避免运行时意外的影响。

问题：为什么泛型是协变的？

协变的泛型符合开发人员的直觉，而这种直接经常是正确的，例如“只读”的一般用法。尽管这种直觉并不总是正确的，Dart 也不需要它总是正确的。Dart 已经选择了最优化的类型检查，所以为什么我们不继续并允许泛型可以协变来传递绕过静态类型检查？

Dart 的类型安全的执行体系允许静态警告以优化而不会带来危险。尽管协变也可能导致问题，但是我们认为那种情况很难发生，而且处理这些问题也会很简单。

我们对语言标记或者暗示不一致的很多种方式都很熟悉。我们认为没有一种现存的方式适合 Dart，所以我们希望类型注释是可选的并且不唐突：强制的标记是不合适的，并且我们感觉推断体系添加了太多的复杂度，得不偿失。

再一次，我们希望更加的实用，而我们认为这样的结果是合理的。

Usage and tools

Q. Should I write my web app in Dart?

We'd love for you to try Dart and give us feedback. If your web app is something that needs to be ultra stable, we recommend waiting until Dart is more mature. However, if you're willing to follow our blog and update your code in response to breaking changes we occasionally make, you may find that Dart is a really exciting community to be a part of!

Q. How does Dart code interoperate with JavaScript libraries?

Although Dart and JavaScript are completely separate languages with potentially separate VMs, they can interoperate. See this article to learn how.

Q. I have a large JavaScript codebase. How would I migrate it to Dart?

Try migrating one major feature at a time, and use the JavaScript interoperability library only when necessary.

Q. I have a large application written in GWT. How do I port it to Dart?

Java and Dart are syntactically similar, so this might be easier than you think. You can rely on Dart Editor to flag any syntax problems. Alternatively, you may consider porting one feature at a time to Dart and using the JavaScript interoperability library as the common middle ground. Be sure to watch our Google I/O 2012 talk Migrating Code from GWT to Dart, but keep in mind that it predates our JavaScript interoperability library.

Q. What browsers do you support as JavaScript compilation targets?

We're currently aiming to support the following browsers:
  • Internet Explorer, versions 9 and 10.
  • Firefox, latest version.
  • Chrome, latest version.
  • Safari for desktop, version 6.

Q. What browsers support Dart now?

As of December 2012, no production browsers can execute Dart code unless it's first compiled to JavaScript. However, the Dart Editor ships with a version of Chromium (called Dartium) that has the Dart VM integrated into it. Dartium not only executes Dart code natively, but it also has really good interoperability with Dart Editor so that you can debug your Dart applications directly from within Dart Editor.

Q. Why was the JavaScript output for "Hello world" so big?

Dart was young. At the launch of Dart, our initial goal was to generate functional JavaScript from Dart code.

Since the initial launch, we've created an improved compiler, dart2js, written in Dart. This compiler aims to be compliant with the language specification, and will continue to be optimized for performance. The dart2js compiler generates significantly smaller JavaScript output than the launch version of dartc (the original Dart-to-JavaScript compiler).

Q. How do I debug Dart code after it's been compiled to JavaScript?

dart2js now generates source maps, so you can debug Dart code in browsers that don't support the Dart VM. However, even in rare cases where you must inspect the raw JavaScript, it's fairly debuggable using the browser's normal JavaScript development tools.

Q. Will any valid Dart code compile to JavaScript, or are there limitations?

Yes. We intend for any valid Dart code to compile to JavaScript. Of course, there are some libraries that will only run on the server because they don't make sense in a browser context.

Q. Does Dart support JSON?

Yes. See the JSON class in the standard library.

Q. Can Dart run on the server?

Yes. The dart:io library is aimed at server-side code that runs on the standalone Dart VM. dart:io uses an asynchronous programming model inspired by node.js, EventMachine, and Twisted. See An Introduction to the dart:io Library to learn more.
